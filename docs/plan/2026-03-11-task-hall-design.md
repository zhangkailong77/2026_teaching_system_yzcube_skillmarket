# 企业任务大厅（独立新项目）设计基线

- 版本: v1.1
- 更新时间: 2026-03-12
- 状态: 已确认，可立项执行
- 关联项目:
  - 学校教学系统（多学校本地部署，既有项目）
  - 企业任务大厅（云端统一部署，独立新仓）

## 1. 项目背景

当前业务存在两类系统：
- 学校侧教学系统: 每个学校一套本地部署，负责账号、课程、考试、进度。
- 平台侧任务大厅: 统一云端一套，负责企业发单、抢单、交付、积分结算。

目标是让多个学校本地教学系统统一接入一个企业任务大厅，实现学员免登录进入任务市场，同时保持学校数据边界。

## 2. 核心决策（本次定稿）

1. 任务大厅使用独立新项目（新仓库），不与教学系统同仓开发。
2. 执行方账号复用学校教学系统账号，通过联邦 SSO 免登录进入任务大厅。
3. 任务大厅内教师账号与学生账号同权，不区分接单能力。
4. 企业可注册登录发单，前置条件为企业资质审核通过。
5. 企业信誉分规则 v1:
   - 初始分 80
   - 分数 < 60 进入人工审核
   - 分数 >= 60 立即恢复自动上架
6. 抢单资格规则 v1:
   - 完成前置课程
   - 当前无进行中任务（或不超过并发上限）
   - 名额未满
7. 能力分权重 v1:
   - 课程完成度 40%
   - 历史任务评分 40%
   - 考试成绩 20%
8. ComfyUI 启动逻辑复用教学系统既有能力，任务大厅先做占位容器。

## 3. 目标与非目标

### 3.1 目标

- 打通“教学系统登录 -> 任务大厅免登录”。
- 上线赏金任务墙、详情、抢单、企业发单的最小闭环。
- 为后续赛道（客服智能体/短视频）预留统一任务框架。

### 3.2 非目标（v1 不做）

- 教学系统与任务大厅账号库完全合并。
- 企业预充值/保证金体系。
- 多组织复杂结算分账。

## 4. 前提条件（必须满足）

### 4.1 学校教学系统侧

1. 每个学校实例具有固定学校配置:
   - `SCHOOL_ID`
   - `SCHOOL_NAME`
2. 学校系统可提供联邦 SSO 出票接口:
   - `POST /api/v1/federation/sso/token`
3. 学校系统与任务大厅网络互通（HTTPS 可访问）。
4. 学校系统与任务大厅约定共享签名验证方式与消费密钥。

### 4.2 任务大厅侧（独立新项目）

1. 能提供 SSO 票据消费回调能力。
2. 能以 `school_id + user_id` 建立跨学校唯一身份映射。
3. 所有执行方业务表必须具备 `school_id` 隔离字段。

### 4.3 联调与运维前提

1. 环境变量管理规范（dev/staging/prod 分离）。
2. Redis 可用（用于一次性票据防重放）。
3. 统一时间同步（NTP），避免短时票据过期误判。

## 5. 总体架构

### 5.1 部署形态

- 教学系统: 多学校本地部署（N 套）
- 任务大厅: 云端统一部署（1 套）

### 5.2 身份流

1. 用户在学校教学系统完成登录。
2. 进入任务大厅时，学校系统签发一次性 SSO ticket（短时有效）。
3. 任务大厅消费 ticket，创建任务大厅会话。
4. ticket 一次性消费后失效（防重放）。

## 6. 跨系统身份与数据映射

### 6.1 用户唯一键

- 推荐唯一键: `global_user_key = school_id:user_id`

### 6.2 数据边界

- 学校系统保留: 密码、课程原始数据、考试原始数据。
- 任务大厅保留: 任务、抢单、交付、信誉分、钱包流水、资格快照。

### 6.3 同步策略

- 能力分相关基础数据可通过事件/定时同步到任务大厅。
- 抢单最终判定在任务大厅执行，必要时可回源学校系统校验。

## 7. 联邦接口契约（v1）

### 7.1 学校系统已实现/待完善

- `POST /api/v1/federation/sso/token`
  - 输入: 当前登录用户上下文
  - 输出: `ticket`, `expires_in`, `school_id`

### 7.2 任务大厅（独立项目需实现）

- `POST /hall/federation/sso/exchange`
  - 输入: `ticket`
  - 处理: 调学校系统消费或本地校验+消费
  - 输出: 任务大厅会话 token/cookie

### 7.3 安全要求

- ticket 含 `typ`, `jti`, `uid`, `sub`, `role`, `school_id`, `exp`
- `jti` 必须一次性消费
- 票据 TTL 建议 30~60 秒
- 消费接口需密钥头（如 `X-Federation-Secret`）

## 8. 业务规则基线

### 8.1 企业信誉分

- 初始 80
- <60 人工审核
- >=60 自动恢复
- 分值区间 0~100
- 所有变更必须写流水

### 8.2 抢单判定顺序

1. 前置课程资格
2. 进行中任务限制
3. 能力分/档位
4. 风控检查
5. 名额原子扣减

### 8.3 能力分公式

`ability_score = course*0.4 + task_quality*0.4 + exam*0.2`

## 9. 独立任务大厅项目（新仓）建议结构

```text
task-hall/
  apps/web/                    # 前端（任务大厅）
  apps/api/                    # 后端（任务域）
  docs/
  deploy/
```

建议页面最小集:
- 登录回调页（SSO）
- 任务墙
- 任务详情（含 ComfyUI 占位容器）
- 企业发单
- 我的作品集
- 积分钱包

## 10. ComfyUI 占位容器规范

1. 仅 `task_type=comfyui` 显示该容器。
2. 先展示静态占位图，不直接嵌入真实面板。
3. 点击“启动实训环境”调用教学系统已有启动能力。
4. 后续替换成真实嵌入时，不改任务业务主流程。

## 11. 实施阶段

### Phase 1（必须）
- 任务大厅新仓初始化
- SSO 回调打通
- 任务墙/详情/抢单
- 企业资质与信誉分
- ComfyUI 占位容器

### Phase 2（应做）
- 作品集
- 钱包流水
- 提现申请
- 审核后台联动

### Phase 3（扩展）
- 客服智能体赛道
- 短视频赛道
- 策略后台配置化

## 12. 风险与应对

1. 风险: 学校网络环境导致联邦接口不稳定
- 应对: 重试机制 + 降级提示 + 健康检查

2. 风险: ticket 重放攻击
- 应对: Redis 一次性消费 + 短 TTL + 服务端密钥校验

3. 风险: 多学校用户冲突
- 应对: 全链路使用 `school_id + user_id` 唯一映射
